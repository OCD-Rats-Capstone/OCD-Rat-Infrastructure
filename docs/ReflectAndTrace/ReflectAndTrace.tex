\subsubsection*{Peer Review Feedback (Team 7)}

\begin{itemize}
    \item \textbf{SRS -- \S 12.1 Speed \& Latency (SAL.R.1, SAL.R.2)}
    \begin{itemize}
        \item \textbf{Feedback:} SAL.R.2 (network requests $<100$ ms) is uncontrollable end-to-end over public networks/FRDR. Recommended scoping to server processing time with percentile SLOs (e.g., p95 $\le$ 400 ms). SAL.R.1 ($\le$ 2 s) lacks scope clarity regarding metadata vs. large media, and needs hardware/concurrency specs for testability.
        \item \textbf{Response \& Changes Made:} Revised SAL.R.2 to focus strictly on server-side API processing latency (p95 $\le$ 400 ms). Clarified SAL.R.1 to apply specifically to metadata queries. Added testable details including dataset size, expected indices, hardware assumptions (4 vCPUs, 16 GB RAM), and concurrency (250 users). Aligned with Hazard Analysis scope.
    \end{itemize}

    \item \textbf{SRS -- \S 6.3 Work Partitioning (NLP Integration)}
    \begin{itemize}
        \item \textbf{Feedback:} Natural language query generation requires strategies to offset LLM limitations and decrease processing time, such as keyword dictionaries, caching, and prompt rulesets.
        \item \textbf{Response \& Changes Made:} Expanded the ``NLP Integration'' subsection. Added explicit mitigation strategies including caching frequent queries, implementing keyword dictionaries for common terms, and designing fallbacks to the structured filter engine to improve accuracy and reduce latency.
    \end{itemize}

    \item \textbf{Hazard Analysis -- \S 4.1 Critical Assumptions}
    \begin{itemize}
        \item \textbf{Feedback:} Treating the FRDR API as an unavoidable point of failure lacks contingencies. Recommended adding alternative data storage forms or caching for project resilience.
        \item \textbf{Response \& Changes Made:} Expanded the ``FRDR API Availability'' assumption. Added discussion of mitigation strategies and contingencies, including local data caching of frequently accessed queries and periodic snapshots to allow limited offline access during FRDR outages.
    \end{itemize}

    \item \textbf{SRS -- \S 2.5 Personas}
    \begin{itemize}
        \item \textbf{Feedback:} Personas rely on generic labels (e.g., ``Undergraduate Student'') and lack personal touches, names, and complete demographics, failing the V.A.R.I.E.D. criteria.
        \item \textbf{Response \& Changes Made:} Revised all system personas to include fictional names (e.g., ``Alex Johnson''), specific ages, locations, and richer demographic details to make the user profiles more actionable and empathetic for the development team.
    \end{itemize}

    \item \textbf{Hazard Analysis -- \S 5.1 Out of Scope Hazards}
    \begin{itemize}
        \item \textbf{Feedback:} The list of out-of-scope hazards lacks justification. Entries like ``user misuses data'' can be mitigated by UI design, and external outages should be explicitly justified as outside team control.
        \item \textbf{Response \& Changes Made:} Added explanatory text to every item in the out-of-scope list. Explicitly justified why external factors (FRDR outages) are beyond the system boundary, and clarified that while UI design mitigates data misinterpretation, the ultimate misuse of data remains an out-of-scope user error.
    \end{itemize}
\end{itemize}

\section{Reflection on Project Management (LO24)}

\plt{This question focuses on processes and tools used for project management.}

\subsection{How Does Your Project Management Compare to Your Development Plan}

\par{We did use the technology we planned on using, basically to a Tee. Python was used for the backend which we somewhat anticipated,
naming python as a technology we would use for 'data analysis and visualizations' but we ended up using it for the entire backend. We anticipated using something like
React which is exactly what we used. Additionally our Database (Postgres), CI/CD (GitHub Actions),
and LLM (ChatGPT) technologies that we anticipated were all what we ended up using more or less.
In terms of our development plan, we did not follow the team member roles at all and we all just acted as general members. We tried to keep up the 
GitHub issues tracker but as the workload of the semester built up, this became more difficult and we took a more ad-hoc approach to feature assignment. Our team
meeting plan was not followed for internal meetings. For meetings with our supervisor, we followed that for the whole semester but the twice weekly internal
team meetings stopped happening after first semester and again were replaced with more ad-hoc meetings. }

\subsection{What Went Well?}

\par{Our DevOps process and technology worked very well. The dockerization of our project made it very easy to run the project on our local machine,
view changes and mimic a production environment in the development phase. Additionally, it was useful for adjusting our project such as updating
our database which only required us to replace the SQL files in the db directory when the project was being built. This also made it very easy to deploy the project
to a remote server as once an actions runner was set up, we could simply run the project build workflow and all the new changes were automatically containerized
and deployed to the server in a very simple and repeatable way.}
\par{Another thing that went well was the use of a React frontend. Using this JavaScript framework gave us access to lots of useful components which made it easy to make
our UI very user friends and asthetically pleasing without wasting a bunch of time on creating our own personal styling components. It also gave us a lot of options in terms of 
building cool visuals as there are many component libraries that provide a wide array of different dynamic front-end functionality.}

\subsection{What Went Wrong?}

\par{Our process for task delegation and code reviews did not go great. This was just a product of 
our general lack of time as university students but we fell behind on assigning explicit tasks for each deliverable
to each team member and everyone just worked on things as they came. This led to disorganized and inconsistent output from members
and made it hard to synchronize our workflows. Additionally, our code standards enforcement was not upheld very well as each PR was supposed to have several
people review it before it was approved. This was more accurately reflected as one person just skims over the PR, not really critiquing it other than glaring issues. In
extreme time crunches, team members would even approve their own PRs on occasion. Finally, our meeting schedule with our supervisor was great but without the external expectations,
our internal meetings were not very frequent like we had initially hoped for and were often not overly productive either.}

\subsection{What Would you Do Differently Next Time?}

\par{Ask more questions early. If we were to restart this project from scratch knowing everything we've learned from our project supervisor
related to what exactly he wants the system to look at as well as the specific nuances of the data, I think we would be able to build something
much more robust and useful. This is easier said than done as we are students with deadlines and other courses so it was quite difficult to take
the time to really carefully elicit what our primary stakeholder was looking for early on. Of course we also did not have intimate knowledge of our 
dataset at that time either so things he thought were self-evident were quite foreign to us. If we were given another chance to do this project with
a more abundant time budget, we would spend more time carefully figuring out what he wants and learning about the nature of the dataset and it's specific applications.
In terms of processes, this might mean more systematic client interactions with more structured questions or orchestrating development sprints to very quickly
elicit feedback out of our supervisor.}

\subsection{Knowledge/Skills Outside of Courses}

\par{Several critical skills and domain knowledge areas were required for this project that extended beyond our standard computer science and engineering curriculum:}

\begin{itemize}
    \item \textbf{Behavioral Neuroscience Domain Knowledge:} Understanding OCD research, rat behavioral trials, spatial-temporal data interpretation, and the scientific context of compulsive behavior metrics. This required extensive collaboration with our client (Dr. Szechtman) and review of primary research literature.
    
    \item \textbf{Large-Scale Data Engineering:} Managing and processing largedatasets, implementing efficient database indexing strategies, designing data pipelines, and optimizing query performance for large-scale research datasets.
    
    \item \textbf{Full-Stack Integration (React + FastAPI + PostgreSQL):} Integrating individual technologies into a production-ready system with proper API design, asynchronous processing, and real-time data handling required hands-on learning beyond coursework.
    
    \item \textbf{DevOps \& Containerization (Docker, GitHub Actions):} Practical containerization, CI/CD pipeline setup, automated testing, and deployment automation were largely self-taught through industry documentation and experimentation.
    
    \item \textbf{External API Integration:} Integrating with the FRDR API for data retrieval and OpenAI/ChatGPT APIs for NLP, including authentication, rate limiting, and error handling.
    
    \item \textbf{LLM Prompt Engineering:} Designing effective prompts, managing LLM contexts, and fine-tuning language models for domain-specific applications required research and iterative testing.
    
    \item \textbf{Research-Oriented Documentation:} Writing user manuals and deployment guides tailored to academic researchers (non-software engineers) required different communication styles than typical technical documentation.
    
    \item \textbf{Client Management \& Requirements Elicitation:} Learning to ask clarifying questions, document implicit assumptions, and manage evolving requirements from domain experts without software engineering backgrounds.
    
    \item \textbf{Video Data Handling:} Managing, streaming, and synchronizing video files with trajectory data, including file I/O optimization.
    
    \item \textbf{Behavioral Analysis Algorithms:} Implementing trajectory smoothing, pattern detection, and compulsive behavior classification required research into signal processing and pattern recognition.
\end{itemize}

\par{Overall, the capstone project required us to bridge the gap between academic computer science and real-world research software engineering, learning how to work effectively with domain experts and manage complex interdisciplinary systems.}
